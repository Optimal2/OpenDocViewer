# AI Agent Documentation

OpenDocViewer keeps a generated agent documentation packet under
`docs-agent/`. It is generated by AgentDocMap from the existing JSDoc comments
and source structure. The generator must not require AgentDocMap-specific
comments in OpenDocViewer source files.

## Agent Read Order

When an AI agent starts work in this repository, use this order:

1. `docs-agent/AGENT_CONTEXT.md`
2. `docs-agent/MODULES.md`
3. `docs-agent/ENTRYPOINTS.md`
4. `docs-agent/DEPENDENCIES.md`
5. `docs-agent/FILE_MAP.md` for the area being changed
6. `docs-agent/SYMBOL_INDEX.md` for JSDoc-backed symbols
7. `docs-src/CODEX_DEVELOPMENT.md` for repository workflow rules

The generated packet is an orientation layer, not a replacement for reading the
source before editing.

## Regenerating

AgentDocMap is expected to be cloned next to OpenDocViewer in local development:

```text
<workspace>/AgentDocMap
<workspace>/OpenDocViewer
```

Regenerate the committed packet with:

```bash
npm run doc:agent
```

The command writes `docs-agent/` with stable metadata so the files can be
committed without changing on every run.

If AgentDocMap is elsewhere:

```bash
npm run doc:agent -- --agentdocmap-root <path-to-AgentDocMap>
```

## Release And CI Expectations

- Update `docs-agent/` when source structure, public JSDoc, package
  dependencies, startup flow, or major module responsibilities change.
- Run `npm run doc:agent` during release preparation.
- The release helper includes `npm run doc:agent` in pre-release validation.
- The GitHub "Agent Documentation" workflow regenerates `docs-agent/` and fails
  if the committed packet is stale.

Do not put private deployment values, customer-specific notes, credentials, or
protected payload examples in generated public agent documentation.
